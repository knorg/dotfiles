* i3 Window Manager
** Philosophy

i3 is a /tiling/ window manager. Windows don't float by default — they
fill the screen and split automatically. Everything is driven by the
keyboard. There is no taskbar, no desktop icons, no minimize button.
You control layout, focus, and workspaces entirely through keybindings.

Your modifier key is =Super= (the Windows key), referred to as =$mod= in the config.

** Core concepts

- *Workspace:* A virtual desktop. You have 10 (keys =1= through =0=). Each
  monitor can show one workspace at a time.
- *Container:* Every window lives in a container. Containers can split
  horizontally or vertically.
- *Tiling:* New windows take half the space of the focused container.
- *Floating:* Some windows (dialogs, popups) float. You can toggle any window
  to floating mode.
- *Scratchpad:* A hidden workspace for stashing windows temporarily.

** Status bar

i3status (or i3blocks) shows system info at the bottom: workspaces on
the left, system tray and status (battery, CPU, date/time) on the right.
Config lives at =~/.config/i3status/config=.

** Practical workflow

1. Boot → you land at a blank screen with the status bar
2. =$mod + Return= → terminal appears, fills the screen
3. =$mod + Return= again → screen splits, two terminals side by side
4. =$mod + d= → Rofi opens → type =firefox= → Enter → browser opens in a new split
5. =$mod + 2= → switch to workspace 2 (empty)
6. =$mod + Return= → terminal on workspace 2
7. =$mod + 1= → back to workspace 1, your windows are still there

* Doom Emacs
** Introduction

Doom Emacs is a configuration framework on top of GNU Emacs. It provides
sane defaults, a module system, and a mnemonic keybinding scheme built
around =SPC= (Space) as the leader key. Your Emacs runs as a *daemon*
(background service) — you connect to it with =emacsclient -c=, which
opens instantly.

*** Starting and stopping

#+BEGIN_SRC bash
# The daemon starts automatically on login (systemd user service)
systemctl --user status emacs       # Check if running
systemctl --user restart emacs      # Restart after major config changes

# Connect to the daemon (what you type day-to-day)
emacsclient -c                      # Open a new frame (window)
emacsclient -c somefile.jl          # Open a file directly

# If the daemon isn't running, start standalone
emacs                               # Runs in foreground (not daemon)
#+END_SRC

*** The SPC leader system

Nearly every command starts with =SPC= (spacebar). Doom shows a popup
(which-key) after a short delay listing available next keys:

| Prefix      | Category               |
|-------------+------------------------|
| =SPC=       | General leader         |
| =SPC f=     | File operations        |
| =SPC b=     | Buffer operations      |
| =SPC w=     | Window management      |
| =SPC p=     | Project operations     |
| =SPC s=     | Search                 |
| =SPC c=     | Code (LSP) actions     |
| =SPC g=     | Git (Magit)            |
| =SPC o=     | Open (REPL, term, etc) |
| =SPC t=     | Toggle settings        |
| =SPC n=     | Notes (Denote, org)    |
| =SPC h=     | Help                   |

*If you ever get lost*, just press =SPC= and wait — the popup tells you
everything available. Press =SPC h b b= to search all keybindings.

*** Doom config files

Your configuration lives in three files:

| File                       | Purpose                                |
|----------------------------+----------------------------------------|
| =~/.config/doom/init.el=   | Enable/disable modules (e.g. =:lang julia=) |
| =~/.config/doom/config.el= | Your personal config (keybinds, faces, settings) |
| =~/.config/doom/packages.el= | Extra packages (e.g. ef-themes, denote) |

After editing =config.el=, run =SPC h r r= (=doom/reload=) to apply changes
without restarting. For =init.el= or =packages.el= changes, you need to
run =doom sync= from the terminal, then restart the daemon:

#+BEGIN_SRC bash
~/.config/emacs/bin/doom sync
systemctl --user restart emacs
#+END_SRC

| Key              | Action                                |
|------------------+---------------------------------------|
| =h / j / k / l=  | Left / down / up / right             |
| =w=              | Jump forward one word                 |
| =b=              | Jump backward one word                |
| =e=              | Jump to end of current word           |
| =0=              | Beginning of line                     |
| =^=              | First non-blank character on line     |

| =gg=             | Top of file                           |
| =G=              | Bottom of file                        |
| =C-d=            | Scroll down half page                 |

** Dired — File Manager

Dired is Emacs' built-in file manager. Open it by pressing =SPC .=
and navigating to a directory, or with =SPC f d=.

*** Navigating

| Key   | Action                               |
|-------+--------------------------------------|
| =j/k= | Move up/down through file list       |
| =RET= | Open file or enter directory          |
| =-=   | Go up to parent directory             |
| =g r= | Refresh directory listing             |
| =(/)==  | Show/hide details (file size, dates) |

*** File operations

| Key   | Action                               |
|-------+--------------------------------------|
| =d=   | Mark file for deletion                |
| =u=   | Unmark one file                       |
| =U=   | Unmark all marks                      |
| =x=   | Execute marked deletions (confirms)   |
| =m=   | Mark file for bulk operation          |
| =R=   | Rename/Move file(s)                   |

To speed up confirmation, add to =config.el=:

=(setq dired-deletion-confirmer #'y-or-n-p)=

This changes the delete prompt from typing =yes= to just pressing =y=.

*** Two-window move workflow

This is the most practical way to move files between directories:

1. =SPC w v= — split into two side-by-side windows
2. In the left window: =SPC .= → navigate to the *source* directory
3. In the right window (=SPC w l= to focus it): =SPC .= → navigate to the *destination*
4. Back in the source (=SPC w h=): mark files with =m=
5. Press =R= — Dired auto-fills the destination from the other window
6. =RET= to confirm the move

The same workflow works with =C= for copying instead of moving.

** Editing Files

*** Modes (evil states)

Doom uses evil-mode, which gives you Vim-style modal editing. You are
always in one of these modes:

| Mode       | How to enter     | What it does                          |
|------------+------------------+---------------------------------------|
| Normal     | =Escape=         | Navigate and execute commands         |
| Insert     | =i=              | Type text at cursor position          |
| Visual     | =v=              | Select text character-by-character    |
| V-Line     | =V=              | Select entire lines                   |
| V-Block    | =C-v=            | Select a rectangular block            |

The mode indicator appears in the cursor shape and the modeline at the
bottom. Always press =Escape= to return to Normal mode before using
any =SPC= commands.

*** Entering Insert mode

differently:

|-------+-------------------------------------------|

| =A=   | Insert at end of line                     |
| =o=   | Open new line below and enter Insert mode |
| =O=   | Open new line above and enter Insert mode |
| =s=   | Delete character and enter Insert mode    |
| =S=   | Delete entire line and enter Insert mode  |

*** Essential editing commands (Normal mode)

| Key       | Action                                    |
|-----------+-------------------------------------------|
| =x=       | Delete character under cursor             |
| =dd=      | Delete entire line                        |
| =D=       | Delete from cursor to end of line         |
| =cc=      | Change entire line (delete + Insert mode) |
| =C=       | Change from cursor to end of line         |
| =yy=      | Yank (copy) entire line                   |
| =p=       | Paste after cursor                        |
| =P=       | Paste before cursor                       |

| =gcc=     | Comment/uncomment current line            |

*** The verb-noun grammar

Vim commands follow a *verb + noun* pattern. This is the most powerful
concept to internalize — once you know a few verbs and nouns, you can
combine them freely.

*Verbs* (what to do):

| Key | Verb   | Meaning                              |
|-----+--------+--------------------------------------|
| =d= | Delete | Remove text and save to register     |
| =c= | Change | Delete text and enter Insert mode     |
| =y= | Yank   | Copy text to register                |
| =>= | Indent | Shift text right                     |
| =<= | Dedent | Shift text left                      |

*Nouns* (what to act on — also called text objects and motions):

| Key    | Noun             | Meaning                       |
|--------+------------------+-------------------------------|
| =w=    | Word             | From cursor to next word      |
| =b=    | Back             | From cursor to previous word  |
| =iw=   | Inner word       | The word under cursor         |
| =aw=   | A word           | Word under cursor + surrounding space |
| =i"=   | Inside quotes    | Text between nearest ="= pair |
| =i(=   | Inside parens    | Text between nearest =()=     |
| =i{=   | Inside braces    | Text between nearest ={}=     |
| =it=   | Inside tag       | Text between HTML/XML tags    |
| =ip=   | Inner paragraph  | The current paragraph         |
| =gg=   | To top           | From cursor to top of file    |
| =G=    | To bottom        | From cursor to bottom of file |
| =3j=   | 3 lines down     | Three lines downward          |

*Examples:*

- =dw= — delete to end of word
- =ci(= — change inside parentheses (delete contents, enter Insert mode)
- =yiw= — yank (copy) the word under cursor
- =di"= — delete inside quotes
- =>ip= — indent the current paragraph
- =y3j= — yank 3 lines down
- =dG= — delete from cursor to end of file
- =gUiw= — uppercase the word under cursor

*A number before any command repeats it:* =3dd= deletes 3 lines, =5j= moves
down 5 lines.

*** Search and replace

| Key              | Action                                 |
|------------------+----------------------------------------|
| =/pattern=       | Search forward                         |
| =?pattern=       | Search backward                        |
| =n= / =N=        | Next / previous match                  |
| =*=              | Search for word under cursor           |
| =SPC s s=        | Search in current buffer (vertico)     |
| =SPC /=          | Search in project (ripgrep)            |
| =SPC s p=        | Search in project (alternative binding)|
| =:%s/old/new/g=  | Replace all in file (command mode)     |
| =:%s/old/new/gc= | Replace all, asking for each match     |

For the replace commands, press =:= first to enter command mode, then
type the pattern. The =c= flag at the end makes it interactive — you
can press =y= to replace, =n= to skip, or =a= to replace all remaining.

*** Visual mode selections

| Key     | Action                                  |
|---------+-----------------------------------------|
| =v=     | Start visual (character) selection      |
| =V=     | Start visual line selection              |
| =C-v=   | Start visual block (column) selection   |
| =gv=    | Reselect last visual selection          |

Once in visual mode, move the cursor to expand the selection, then
apply a verb: =d= to delete, =y= to yank, =>= to indent, =gc= to

** Magit — Git from Emacs

Magit is a complete Git interface inside Emacs. It replaces all
command-line Git usage with a keyboard-driven buffer.

*** Opening Magit

| Key       | Action                     |
|-----------+----------------------------|
| =SPC g g= | Open Magit status (main hub) |
| =SPC g B= | Blame current file         |
| =SPC g l= | Git log                    |

*** The status buffer

After =SPC g g=, you see sections for unstaged changes, staged changes,
recent commits, and branch info. Navigate with =j= / =k= and press
=TAB= to expand/collapse sections.

| Key         | Action                                 |
|-------------+----------------------------------------|
| =s=         | Stage file or hunk under cursor        |
| =u=         | Unstage file or hunk                   |
| =TAB=       | Expand/collapse section or hunk        |
| =?=         | Show all available commands (always works) |
| =q=         | Close current Magit buffer             |
| =g=         | Refresh status                         |

*** Committing

| Key       | Action                                       |
|-----------+----------------------------------------------|
| =c c=     | Start a new commit (opens message buffer)    |
| =c a=     | Amend last commit (edit message + add staged) |
| =c e=     | Extend last commit (add staged, keep message) |
| =c w=     | Reword last commit message only              |

In the commit message buffer:

| Key         | Action              |
|-------------+---------------------|
| =C-c C-c=   | Confirm commit     |
| =C-c C-k=   | Abort commit       |
| =, ,=       | Confirm (Doom shortcut) |
| =, k=       | Abort (Doom shortcut)   |
| =M-p / M-n= | Cycle through previous commit messages |

The first line is the subject (keep under 50 characters). Leave a blank
line, then write the extended description body.

*** Hunk-level staging

You don't have to stage entire files. Expand a file with =TAB= to see
individual hunks (changed sections). Put the cursor on a specific hunk
and press =s= to stage just that hunk. This lets you split a file's
changes across multiple precise commits.

*** Pushing and pulling

| Key   | Action                    |
|-------+---------------------------|
| =P p= | Push to remote            |
| =F p= | Pull (fetch + merge)      |
| =f f= | Fetch only (no merge)     |

*** Branching

| Key   | Action                    |
|-------+---------------------------|
| =b b= | Checkout existing branch  |
| =b c= | Create and checkout new branch |
| =b d= | Delete branch             |

*** Undoing commits (reset)

Move your cursor to the *branch/HEAD line* at the top of the status
buffer (not on a file), then:

| Command        | Effect                                    |
|----------------+-------------------------------------------|
| =X s HEAD~1=   | Soft reset — undo commit, keep staged     |
| =X m HEAD~1=   | Mixed reset — undo commit, keep unstaged  |
| =X h HEAD~1=   | Hard reset — undo commit, DELETE changes  |

#+BEGIN_QUOTE
Important: =X= on a *file* means "untrack" — make sure your cursor is
on the HEAD line before pressing =X= for reset.
#+END_QUOTE

** Julia IDE

Your Doom config enables Julia with LSP and tree-sitter. The relevant
modules in =init.el= are:

#+BEGIN_SRC elisp
;; :tools
(lsp +eglot)              ; Language server protocol via eglot
tree-sitter               ; Better syntax highlighting

;; :lang
(julia +lsp +tree-sitter) ; Julia with LSP and tree-sitter

;; :term
vterm                     ; Terminal emulator for REPL
#+END_SRC

When you open a =.jl= file, eglot (the LSP client) starts the Julia
language server automatically. *First launch is slow* — Julia has to
precompile =LanguageServer.jl=. Subsequent starts are much faster.

*** Prerequisites

#+BEGIN_SRC bash
# System dependencies (installed by setup script)
sudo apt install cmake libvterm-dev libtool-bin

# Julia package (must be installed once in Julia itself)
julia -e 'using Pkg; Pkg.add("LanguageServer")'

# Verify it loads
julia -e 'using LanguageServer'
#+END_SRC

*** Completion

Completion triggers automatically as you type (powered by corfu).

| Key       | Action                         |
|-----------+--------------------------------|
| (auto)    | Popup appears as you type      |
| =C-SPC=   | Manually trigger completion    |
| =C-n=     | Next completion item           |

*** REPL workflow

| Key       | Action                              |
|-----------+-------------------------------------|
| =SPC o r= | Open Julia REPL (vterm)             |
| =C-c C-c= | Send current line or selection      |
| =C-c C-b= | Send entire buffer to REPL          |
| =C-c C-d= | Send current function definition    |
| =C-c C-z= | Switch between source and REPL      |

The REPL runs in a vterm buffer at the bottom. You write code in the
top buffer and send expressions down to the REPL for evaluation. This
is the standard Julia interactive workflow — write, send, check, iterate.

*** Typical session

1. =SPC f f= → open =myproject/src/main.jl=
2. =SPC o r= → Julia REPL appears at the bottom
3. Write a function in the top buffer
4. =C-c C-d= → sends the function to REPL (defined)
5. Switch to REPL: =C-c C-z= → call the function, test it interactively
6. =C-c C-z= → back to source, edit the function
7. =C-c C-c= → re-send the changed code
8. Errors show inline via LSP diagnostics (red underlines, =] d= to jump)

To close the REPL buffer: =SPC b k= or =q= in normal mode.

** Denote — Note Taking

Denote is a minimalist note-taking package. Its core principle is that
notes are just files with structured filenames — no database, no
proprietary format, no lock-in. Everything is a plain org file that
you can grep, move, or edit with any tool.

*** Installation (already in your config)

=packages.el=:
#+BEGIN_SRC elisp
(package! denote)
(package! consult-denote)  ; optional: live preview for search
#+END_SRC

*** How Denote names files

Every note gets an automatic filename with three parts:

#+BEGIN_EXAMPLE
20260301T143022--meeting-notes-on-project-x__work_julia.org
│                │                             │
│                │                             └─ keywords (tags)
│                └─ title (slugified with hyphens)
└─ timestamp (permanent unique ID, never changes)
#+END_EXAMPLE

The timestamp is the note's identity. Even if you rename the title or
change keywords later, the timestamp stays the same. This means links
between notes never break when you rename things.

Keywords are separated by underscores. A note can have zero or many
keywords. They act as lightweight tags for filtering and searching.

*** Where notes live

All notes go into =~/notes/=. Denote uses a flat directory by default —
no subdirectories needed. Organization comes from keywords and search,
not from folder hierarchies.

You can change this in =config.el=:

#+BEGIN_SRC elisp
(setq denote-directory "~/notes/")
#+END_SRC

*** Creating notes

=SPC n d n= starts the new-note wizard. Denote prompts you step by step:

1. *Title:* Type the note title in plain language (e.g. "Meeting notes on Project X")
2. *Keywords:* Type tags separated by commas (e.g. "work, julia"). Tab-completes
   against previously used keywords. Press =RET= with empty input to skip.
3. A new buffer opens with front matter already filled in. Start writing.

The file is immediately saved with the structured filename. The front
matter in the org file looks like:

#+BEGIN_EXAMPLE
#+title:      Meeting notes on Project X
#+date:       [2026-03-01 Sat 14:30]
#+filetags:   :work:julia:
#+identifier: 20260301T143022
#+END_EXAMPLE

*** Finding and searching notes

| Key         | Action                                     |
|-------------+--------------------------------------------|
| =SPC n d f= | Find note by filename (fuzzy, live preview)|
| =SPC n d g= | Grep across all note *contents*            |

=SPC n d f= uses the filename components (timestamp, title, keywords)
for fuzzy matching. With =consult-denote= installed, you get a live
preview of each note as you move through the results.

=SPC n d g= searches inside the text of all notes — useful when you
remember a phrase but not which note it was in.

You can also use the general project search: open =~/notes/= as a
project (=SPC p p=, then select it) and use =SPC /= to ripgrep across
all notes.

*** Linking between notes

| Key         | Action                               |
|-------------+--------------------------------------|
| =SPC n d l= | Insert a link to another note        |
| =SPC n d b= | Show backlinks (notes linking here)  |

When you press =SPC n d l=, Denote prompts you to select a note (fuzzy
search). It inserts an org-mode link using the timestamp as the
identifier:

#+BEGIN_EXAMPLE
[[denote:20260301T143022][Meeting notes on Project X]]
#+END_EXAMPLE

Follow the link by placing your cursor on it and pressing =RET= (or =gd=
in evil normal mode).

*Backlinks* (=SPC n d b=) are the reverse: they show you all other notes
that link /to/ the current note. This is how you build a web of
connected notes without maintaining an index manually. Over time, your
notes develop an organic structure through these links — related ideas
cluster together naturally.

*** Renaming notes

| Key         | Action                                    |
|-------------+-------------------------------------------|
| =SPC n d r= | Rename current note (title and keywords)  |

Denote re-prompts you for title and keywords, then renames the file.
The timestamp never changes. Links from other notes still work because
they reference the timestamp, not the title.

*** Practical note-taking patterns

*Meeting notes:* =SPC n d n= → title "Sprint planning 2026-03-03" →
keywords "work, meetings". Add TODO items for action points. Set
deadlines so they appear in org-agenda.

*Learning journal:* =SPC n d n= → title "Julia metaprogramming notes" →
keywords "julia, learning". Link to related notes as you go.

*Project log:* Create one note per project. Use linking to connect
decisions to meeting notes. Use keywords consistently (e.g. always
"projectx") to filter all related notes with =SPC n d f=.

*Quick capture:* Even a one-line note is fine. Denote creates the file
instantly — no friction. You can expand it later or let it stay small.

** Org-mode

Org-mode is Emacs' built-in system for writing, outlining, task
management, and literate programming. Every Denote note you create

much more powerful.

Org-mode activates automatically in any =.org= file.

*** Headlines and folding

Org files are structured with headlines. Headlines start with one or
more stars (=*=). The number of stars indicates the depth level. You can
fold and unfold sections to navigate large documents — this is one of
org-mode's most practical features.

| Key             | Action                              |
|-----------------+-------------------------------------|
| =TAB=           | Fold/unfold current heading         |
| =Shift-TAB=     | Cycle global fold state             |
| =z a=           | Toggle fold of current subtree (evil) |

=Shift-TAB= cycles through three states:
1. *Overview* — only top-level headings visible
2. *Contents* — all headings visible, bodies hidden
3. *Show all* — everything expanded

** Org-agenda

Org-agenda is a unified view of your tasks, deadlines, and scheduled
items pulled from across your org files. It aggregates everything into
a single actionable view — your personal command center for what needs
doing and when.

*** Setting up agenda files

The agenda needs to know which files to scan. Add to =config.el=:

#+BEGIN_SRC elisp
(setq org-agenda-files '("~/notes/"))
#+END_SRC

This tells org-agenda to scan all =.org= files in =~/notes/=. Since
your Denote notes are org files in =~/notes/=, any TODO items or
deadlines you add in notes automatically appear in the agenda.

You can include multiple directories and specific files:

#+BEGIN_SRC elisp
(setq org-agenda-files '("~/notes/" "~/org/todo.org" "~/org/work.org"))
#+END_SRC

*** Opening the agenda

| Key         | Action                              |
|-------------+-------------------------------------|
| =SPC o A=   | Open agenda dispatcher              |
| =SPC o a=   | Open default agenda view            |

The dispatcher (=SPC o A=) presents these built-in views:

| Key | View                                           |
|-----+------------------------------------------------|
| =a= | Weekly agenda + scheduled/deadline items        |
| =t= | Global TODO list (all TODOs across files)       |
| =T= | TODO list filtered by specific keyword          |
| =s= | Full-text search across agenda files            |
| =m= | Match items by tags                             |

*** The weekly agenda view

=SPC o A a= opens the most useful default: a week calendar showing
your scheduled tasks, deadlines, and timestamps.

What you see:
- Today's date highlighted
- Scheduled tasks on their scheduled date
- Deadlines with countdown warnings (e.g. "In 3 d.")
- Overdue items flagged at the top
- Completed items marked with =DONE=

*** Navigating the agenda

| Key         | Action                                |
|-------------+---------------------------------------|
| =j / k=     | Move up/down between items            |
| =RET=       | Jump to the item in its source file   |
| =TAB=       | Show item in other window (stay in agenda) |
| =f=         | Forward one day/week                  |
| =b=         | Backward one day/week                 |
| =.=         | Jump to today                         |
| =v d=       | Switch to day view                    |
| =v w=       | Switch to week view                   |
| =v m=       | Switch to month view                  |
| =r=         | Rebuild/refresh the agenda            |
| =q=         | Quit agenda                           |

*** Acting on items from the agenda

You can modify tasks directly from the agenda without opening the
source file:

| Key         | Action                                    |
|-------------+-------------------------------------------|
| =t=         | Cycle TODO state (=TODO= → =DONE= → none) |
| =SPC m d=   | Set or change deadline                    |
| =SPC m s=   | Set or change scheduled date              |
| =:=         | Set tags                                   |
| =,=         | Set priority (A/B/C)                       |
| =I=         | Clock in (start tracking time)             |
| =O=         | Clock out (stop tracking time)             |

*** Filtering

While in any agenda view, narrow down what's shown:

| Key   | Action                                       |
|-------+----------------------------------------------|
| =/=   | Filter by tag                                |
| =<=   | Filter by category (based on filename)       |
| ===   | Filter by regular expression                 |
| =|=   | Remove all active filters                    |

*** The TODO list view

=SPC o A t= shows a flat list of all TODO items across your agenda
files, regardless of dates. This gives you a complete task overview.
Items are grouped by file.

*** Tag matching view

=SPC o A m= lets you filter by tag expressions. Examples:

- =work= — all items tagged =:work:=
- =work+urgent= — items tagged with both =:work:= and =:urgent:=
- =work-meeting= — items tagged =:work:= but NOT =:meeting:=
- =work|personal= — items tagged =:work:= OR =:personal:=

*** Custom agenda commands

Define your own views in =config.el= for one-key access. Example — a
daily dashboard showing today's schedule plus high-priority tasks:

#+BEGIN_SRC elisp
(setq org-agenda-custom-commands
      '(("d" "Dashboard"
         ((agenda "" ((org-agenda-span 'day)))
          (todo "TODO"
                ((org-agenda-overriding-header "Active TODOs")
                 (org-agenda-sorting-strategy '(priority-down))))))
        ("w" "Work tasks"
         ((tags-todo "work"
                     ((org-agenda-overriding-header "Work TODOs")))
          (agenda "" ((org-agenda-span 'week)
                      (org-agenda-tag-filter-preset '("+work"))))))))
#+END_SRC

Access these with =SPC o A d= (dashboard) or =SPC o A w= (work view).

*** Making Denote and agenda work together

Since your Denote notes live in =~/notes/= and are org files, they
integrate naturally with org-agenda:

1. Set =org-agenda-files= to include =~/notes/=
2. Add TODO items and deadlines in your Denote notes as normal
3. =SPC o A a= — tasks from all notes appear in the weekly view

This means a note about a meeting can contain TODO actions with
deadlines, and those actions show up in your agenda alongside
everything else. No separate task manager needed — your notes /are/
your task manager.

*** Agenda workflow example

1. =SPC n d n= → create "Sprint planning review" with keyword "work"
2. Inside the note, add TODO headings for each action item
3. Set deadlines: =SPC m d= on each TODO heading
4. Later: =SPC o A a= → your deadlines appear in the weekly view
5. In the agenda, press =t= on a task to mark it =DONE=
6. Press =RET= to jump to the note if you need to add details
7. =SPC o A d= → check your dashboard for the big picture

*** REPL — iron.nvim (known issue)

Your config includes iron.nvim for REPL integration with Revise.jl
support. The REPL is configured to launch Julia with the =-i= flag
for interactive mode and a startup expression that loads Revise.

*Current status: the REPL is not yet working reliably.*

The issue is the REPL closing immediately after opening. The =-i= flag
was added to keep Julia interactive after executing the startup
expression, which partially addressed it.

**** Troubleshooting steps

1. *Test Julia directly* — make sure Julia itself works:

   #+BEGIN_SRC bash
   julia -i -e 'using Revise'
   #+END_SRC

   If this fails, fix Revise first:

   #+BEGIN_SRC bash
   julia -e 'using Pkg; Pkg.add("Revise")'
   #+END_SRC

2. *Check iron.nvim config* — open Neovim and run:

   #+BEGIN_SRC vim
   :IronRepl
   #+END_SRC

   If it closes immediately, check =:messages= for errors.

3. *Verify the REPL command* — in your nvim config, iron.nvim should have:

   #+BEGIN_SRC lua
   repl_definition = {
     julia = {
       command = { "julia", "-i", "-e", 'using Revise' },
     },
   },
   #+END_SRC

   The =-i= flag is critical. Without it, Julia exits after =-e=.

4. *Try without Revise* — to isolate the problem:

   #+BEGIN_SRC lua
   command = { "julia", "-i" },
   #+END_SRC

   If this works, wrap Revise in a try/catch:

   #+BEGIN_SRC lua
   command = { "julia", "-i", "-e", 'try using Revise catch e @warn "Revise not loaded" end' },
   #+END_SRC

5. *Test Neovim's terminal directly* — bypass iron.nvim:

   #+BEGIN_SRC vim
   :terminal julia -i
   #+END_SRC

   If Julia stays open here, the issue is in iron.nvim's configuration,
   not in Neovim's terminal.

**** Expected workflow (once working)

| Key           | Action                             |
|---------------+------------------------------------|
| =<leader>rs=  | Start REPL                         |
| =<leader>rc=  | Send current line to REPL          |
| =<leader>rf=  | Send file to REPL                  |
| =<leader>rv=  | Send visual selection to REPL      |

**** Alternative: use a split terminal

#+BEGIN_SRC vim

#+END_SRC

To interact with the terminal: press =i= to enter Terminal mode
(type into Julia), then =C-\= =C-n= to exit back to Normal mode
