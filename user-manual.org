* i3 Window Manager
** Philosophy

i3 is a /tiling/ window manager. Windows don't float by default — they
fill the screen and split automatically. Everything is driven by the
keyboard. There is no taskbar, no desktop icons, no minimize button.
You control layout, focus, and workspaces entirely through keybindings.

Your modifier key is =Super= (the Windows key), referred to as =$mod= in the config.

** Core concepts

- *Workspace:* A virtual desktop. You have 10 (keys =1= through =0=). Each
  monitor can show one workspace at a time.
- *Container:* Every window lives in a container. Containers can split
  horizontally or vertically.
- *Tiling:* New windows take half the space of the focused container.
- *Floating:* Some windows (dialogs, popups) float. You can toggle any window
  to floating mode.
- *Scratchpad:* A hidden workspace for stashing windows temporarily.

** Status bar

i3status (or i3blocks) shows system info at the bottom: workspaces on
the left, system tray and status (battery, CPU, date/time) on the right.
Config lives at =~/.config/i3status/config=.

** Practical workflow

1. Boot → you land at a blank screen with the status bar
2. =$mod + Return= → terminal appears, fills the screen
3. =$mod + Return= again → screen splits, two terminals side by side
4. =$mod + d= → Rofi opens → type =firefox= → Enter → browser opens in a new split
5. =$mod + 2= → switch to workspace 2 (empty)
6. =$mod + Return= → terminal on workspace 2
7. =$mod + 1= → back to workspace 1, your windows are still there

* Doom Emacs
** Introduction

Doom Emacs is a configuration framework on top of GNU Emacs. It provides
sane defaults, a module system, and a mnemonic keybinding scheme built
around =SPC= (Space) as the leader key. Your Emacs runs as a *daemon*
(background service) — you connect to it with =emacsclient -c=, which
opens instantly.

*** Starting and stopping

#+BEGIN_SRC bash
# The daemon starts automatically on login (systemd user service)
systemctl --user status emacs       # Check if running
systemctl --user restart emacs      # Restart after major config changes

# Connect to the daemon (what you type day-to-day)
emacsclient -c                      # Open a new frame (window)
emacsclient -c somefile.jl          # Open a file directly

# If the daemon isn't running, start standalone
emacs                               # Runs in foreground (not daemon)
#+END_SRC

| File                         | Purpose                                          |
|------------------------------+--------------------------------------------------|
| =~/.config/doom/init.el=     | Enable/disable modules (e.g. =:lang julia=)      |

|-------+--------------------------------------|

*** Two-window move workflow

This is the most practical way to move files between directories:

1. =SPC w v= — split into two side-by-side windows
2. In the left window: =SPC .= → navigate to the *source* directory
3. In the right window (=SPC w l= to focus it): =SPC .= → navigate to the *destination*
4. Back in the source (=SPC w h=): mark files with =m=
5. Press =R= — Dired auto-fills the destination from the other window
6. =RET= to confirm the move

The same workflow works with =C= for copying instead of moving.

*** The verb-noun grammar

Vim commands follow a *verb + noun* pattern. This is the most powerful
concept to internalize — once you know a few verbs and nouns, you can
combine them freely.

*Verbs* (what to do):

| Key | Verb   | Meaning                              |
|-----+--------+--------------------------------------|
| =d= | Delete | Remove text and save to register     |
| =c= | Change | Delete text and enter Insert mode     |
| =y= | Yank   | Copy text to register                |
| =>= | Indent | Shift text right                     |
| =<= | Dedent | Shift text left                      |

*Nouns* (what to act on — also called text objects and motions):

| Key    | Noun             | Meaning                       |
|--------+------------------+-------------------------------|
| =w=    | Word             | From cursor to next word      |
| =b=    | Back             | From cursor to previous word  |
| =iw=   | Inner word       | The word under cursor         |
| =aw=   | A word           | Word under cursor + surrounding space |
| =i"=   | Inside quotes    | Text between nearest ="= pair |
| =i(=   | Inside parens    | Text between nearest =()=     |
| =i{=   | Inside braces    | Text between nearest ={}=     |
| =it=   | Inside tag       | Text between HTML/XML tags    |
| =ip=   | Inner paragraph  | The current paragraph         |
| =gg=   | To top           | From cursor to top of file    |
| =G=    | To bottom        | From cursor to bottom of file |
| =3j=   | 3 lines down     | Three lines downward          |

*Examples:*

- =dw= — delete to end of word
- =ci(= — change inside parentheses (delete contents, enter Insert mode)
- =yiw= — yank (copy) the word under cursor
- =di"= — delete inside quotes
- =>ip= — indent the current paragraph
- =y3j= — yank 3 lines down
- =dG= — delete from cursor to end of file
- =gUiw= — uppercase the word under cursor

*A number before any command repeats it:* =3dd= deletes 3 lines, =5j= moves
down 5 lines.

|------------------+----------------------------------------|
| =/pattern=       | Search forward                         |

| Key       | Action                                       |
|-----------+----------------------------------------------|

You don't have to stage entire files. Expand a file with =TAB= to see
individual hunks (changed sections). Put the cursor on a specific hunk
and press =s= to stage just that hunk. This lets you split a file's
changes across multiple precise commits.

*** Undoing commits (reset)

Move your cursor to the *branch/HEAD line* at the top of the status
buffer (not on a file), then:

| Command        | Effect                                    |
|----------------+-------------------------------------------|
| =X s HEAD~1=   | Soft reset — undo commit, keep staged     |
| =X m HEAD~1=   | Mixed reset — undo commit, keep unstaged  |
| =X h HEAD~1=   | Hard reset — undo commit, DELETE changes  |

#+BEGIN_QUOTE
Important: =X= on a *file* means "untrack" — make sure your cursor is
on the HEAD line before pressing =X= for reset.
#+END_QUOTE

** Julia IDE

Your Doom config enables Julia with LSP and tree-sitter. The relevant
modules in =init.el= are:

#+BEGIN_SRC elisp
;; :tools
(eval +overlay)           ; REPL support (SPC o r, C-c C-c send)

;; :lang

When you open a =.jl= file, eglot (the LSP client) starts the Julia

*** REPL workflow

The Julia REPL uses the =julia-repl= package with a vterm backend
(configured in your =config.el=). Doom's =(eval +overlay)= module
provides the send-to-REPL functionality.

| Key       | Action                              |
|-----------+-------------------------------------|
| =SPC o r= | Open Julia REPL (vterm)             |
| =C-c C-c= | Send current line or selection      |
| =C-c C-b= | Send entire buffer to REPL          |
| =C-c C-d= | Send current function definition    |
| =C-c C-z= | Switch between source and REPL      |

The REPL runs in a vterm buffer at the bottom. You write code in the
top buffer and send expressions down to the REPL for evaluation. This
is the standard Julia interactive workflow — write, send, check, iterate.

*** Typical session

1. =SPC f f= → open =myproject/src/main.jl=
2. =SPC o r= → Julia REPL appears at the bottom
3. Write a function in the top buffer
4. =C-c C-d= → sends the function to REPL (defined)
5. Switch to REPL: =C-c C-z= → call the function, test it interactively
6. =C-c C-z= → back to source, edit the function
7. =C-c C-c= → re-send the changed code
8. Errors show inline via LSP diagnostics (red underlines, =] e= to jump)

To close the REPL buffer: =SPC b k= or =q= in normal mode.

** Denote — Note Taking

Denote is a minimalist note-taking package. Its core principle is that
notes are just files with structured filenames — no database, no
proprietary format, no lock-in. Everything is a plain org file that
you can grep, move, or edit with any tool.

*** Installation (already in your config)

=packages.el=:
#+BEGIN_SRC elisp
(package! denote)
(package! consult-denote)  ; optional: live preview for search
#+END_SRC

*** How Denote names files

Every note gets an automatic filename with three parts:

#+BEGIN_EXAMPLE
20260301T143022--meeting-notes-on-project-x__work_julia.org
│                │                             │
│                │                             └─ keywords (tags)
│                └─ title (slugified with hyphens)
└─ timestamp (permanent unique ID, never changes)
#+END_EXAMPLE

The timestamp is the note's identity. Even if you rename the title or
change keywords later, the timestamp stays the same. This means links
between notes never break when you rename things.

Keywords are separated by underscores. A note can have zero or many
keywords. They act as lightweight tags for filtering and searching.

*** Where notes live

All notes go into =~/Notes/= (capital N). Denote uses a flat directory
by default — no subdirectories needed. Organization comes from keywords
and search, not from folder hierarchies. Your config also supports
creating notes in subdirectories via =SPC n d s=.

Your =config.el=:

#+BEGIN_SRC elisp
(setq denote-directory (expand-file-name "~/Notes/")
      denote-file-type 'org
      denote-prompts '(title keywords))
(denote-rename-buffer-mode 1)
#+END_SRC

*** Creating notes

=SPC n d n= starts the new-note wizard. Denote prompts you step by step:

1. *Title:* Type the note title in plain language (e.g. "Meeting notes on Project X")
2. *Keywords:* Type tags separated by commas (e.g. "work, julia"). Tab-completes
   against previously used keywords. Press =RET= with empty input to skip.
3. A new buffer opens with front matter already filled in. Start writing.

The file is immediately saved with the structured filename. The front
matter in the org file looks like:

#+BEGIN_EXAMPLE
#+title:      Meeting notes on Project X
#+date:       [2026-03-01 Sat 14:30]
#+filetags:   :work:julia:
#+identifier: 20260301T143022
#+END_EXAMPLE

*** Finding and searching notes

| Key         | Action                                     |
|-------------+--------------------------------------------|
| =SPC n d o= | Open or create note (fuzzy, live preview)  |
| =SPC n d g= | Grep across all note *contents*            |

=SPC n d o= (=denote-open-or-create=) searches your notes by filename
components (timestamp, title, keywords). If no match is found, it
offers to create a new note with your search term as the title. With
=consult-denote= active, you get a live preview of each note as you
move through the results.

=SPC n d g= searches inside the text of all notes — useful when you
remember a phrase but not which note it was in.

You can also use the general project search: open =~/Notes/= as a
project (=SPC p p=, then select it) and use =SPC /= to ripgrep across
all notes.

*** Linking between notes

| Key         | Action                                   |
|-------------+------------------------------------------|
| =SPC n d i= | Insert link to an existing note          |
| =SPC n d l= | Link or create (insert link, create if new) |
| =SPC n d b= | Show backlinks (notes linking here)      |
| =SPC n d B= | Find backlink (select and jump to it)    |

=SPC n d i= (=denote-link=) prompts you to select an existing note
(fuzzy search) and inserts an org link using the timestamp:

#+BEGIN_EXAMPLE
[[denote:20260301T143022][Meeting notes on Project X]]
#+END_EXAMPLE

=SPC n d l= (=denote-link-or-create=) does the same but if the note
doesn't exist yet, it creates a new one and links to it — useful for
building a web of notes on the fly.

Follow any link by placing your cursor on it and pressing =RET= (or
=gd= in evil normal mode).

*Backlinks* (=SPC n d b=) show all other notes that link /to/ the
current note. =SPC n d B= (=denote-find-backlink=) lets you pick one
and jump to it directly. Over time, your notes develop an organic
structure through these links — related ideas cluster together
naturally.

*** Renaming and managing notes

| Key         | Action                                    |
|-------------+-------------------------------------------|
| =SPC n d r= | Rename current note (title and keywords)  |
| =SPC n d k= | Add keywords to current note              |
| =SPC n d K= | Remove keywords from current note         |
| =SPC n d s= | Create note in a subdirectory             |

** Org-mode

Org-mode is Emacs' built-in system for writing, outlining, task
management, and literate programming. Every Denote note you create
is an org file by default, so understanding org-mode makes your notes
much more powerful.

Org-mode activates automatically in any =.org= file. Your org-directory
is set to =~/org/= (for standalone org files outside Denote).

*** Headlines and folding

Org files are structured with headlines. Headlines start with one or
more stars (=*=). The number of stars indicates the depth level. You can
fold and unfold sections to navigate large documents — this is one of
org-mode's most practical features.

| Key             | Action                              |
|-----------------+-------------------------------------|
| =TAB=           | Fold/unfold current heading         |
| =Shift-TAB=     | Cycle global fold state             |
| =z a=           | Toggle fold of current subtree (evil) |

=Shift-TAB= cycles through three states:
1. *Overview* — only top-level headings visible
2. *Contents* — all headings visible, bodies hidden
3. *Show all* — everything expanded

*** TODO items

Any headline can become a task by adding a TODO keyword at the start.
Your config defines these states:

| State        | Key | Meaning                           |
|--------------+-----+-----------------------------------|
| =TODO=       | =t= | Task to be done                   |
| =NEXT=       | =n= | Active task, working on it now    |

TODO items can have a priority level that affects their sort order
in the agenda:

| Key       | Action                              |
|-----------+-------------------------------------|
| =SPC m p= | Set priority (A/B/C) on heading     |

Priorities are written as =[#A]=, =[#B]=, or =[#C]= after the TODO keyword.
=[#A]= is the highest priority. In the agenda view, press =,= to change
priority on the selected item.

everything beneath it.

Org-agenda is a unified view of your tasks, deadlines, and scheduled

*** Setting up agenda files (already configured)

Your =config.el= scans both your org directory and Denote notes:

The actual config:

#+BEGIN_SRC elisp
(setq org-agenda-custom-commands
      '(("d" "Dashboard"
         ((agenda "" ((org-agenda-span 1)
                      (org-agenda-overriding-header "Today")))
          (todo "NEXT"   ((org-agenda-overriding-header "Next Actions")))
          (todo "WAITING" ((org-agenda-overriding-header "Waiting On")))))

        ("w" "Weekly Review"
         ((agenda "" ((org-agenda-span 7)))
          (todo "TODO" ((org-agenda-overriding-header "All TODOs")))
          (todo "WAITING" ((org-agenda-overriding-header "Waiting On")))))))
#+END_SRC

Access these with =SPC o A= then =d= (dashboard) or =w= (weekly review).

*** Making Denote and agenda work together

Since your Denote notes live in =~/Notes/= and your =org-agenda-files=
already includes that directory, they integrate automatically:

1. TODO items and deadlines in any Denote note appear in the agenda
2. =SPC o A a= — weekly view includes tasks from all notes
3. =SPC o A d= — dashboard shows today's items + NEXT actions

This means a note about a meeting can contain TODO actions with
deadlines, and those actions show up in your agenda alongside
everything else. No separate task manager needed — your notes /are/
your task manager.

*** Agenda workflow example

1. =SPC n d n= → create "Sprint planning review" with keyword "work"
2. Inside the note, add headings with TODO or NEXT items
3. Set deadlines: =SPC m d= on each TODO heading
4. Later: =SPC o A= then =d= → Dashboard shows today + NEXT + WAITING
5. Press =t= on a task → fast-select =d= to mark it =DONE=
6. Press =RET= to jump to the note if you need to add details
7. End of week: =SPC o A= then =w= → Weekly Review for the big picture

** mini.ai — Enhanced Text Objects

mini.ai extends Vim's built-in text objects with more targets and
the ability to seek to the next/previous occurrence. It works with
all verbs (=d=, =c=, =y=, =v=, etc.).

*** Standard text objects (enhanced)

| Text object | What it selects                   |
|-------------+-----------------------------------|
| =i)= / =a)==  | Inside / around parentheses      |
| =i]= / =a]=   | Inside / around brackets         |
| =i}= / =a}=   | Inside / around braces           |
| =i"= / =a"=   | Inside / around double quotes    |
| =i'= / =a'=   | Inside / around single quotes    |
| =i`= / =a`=   | Inside / around backticks        |
| =it= / =at=   | Inside / around HTML/XML tags    |

*** Next / last seeking

The key feature of mini.ai is automatic seeking: if the cursor is
not currently inside the text object, it seeks forward to the *next*
occurrence. Use the =n= (next) and =l= (last/previous) modifiers:

| Key     | Action                                     |
|---------+--------------------------------------------|
| =cin)=  | Change inside the [n]ext parentheses       |
| =cil)=  | Change inside the [l]ast parentheses       |
| =dinq=  | Delete inside the [n]ext [q]uote           |
| =yinq=  | Yank inside the next quote                 |

This means you don't have to move your cursor to the target first —
=cin"= from anywhere on the line will find and change inside the next
double quotes.

=n_lines = 500= in your config means mini.ai searches up to 500 lines
for text objects.

*** Examples

- =va)= — [V]isually select [A]round [)]parentheses
- =yinq= — [Y]ank [I]nside [N]ext [Q]uote
- =ci'= — [C]hange [I]nside =[']= quotes
- =daa= — [D]elete [A]round argument (function parameter)

** mini.surround — Add/Delete/Replace Surroundings

mini.surround lets you manipulate surrounding characters (brackets,
quotes, tags) with three operations. All use the =s= prefix:

| Key       | Action                              | Example                   |
|-----------+-------------------------------------+---------------------------|
| =sa{motion}{char}= | [S]urround [A]dd            | =saiw)= → surround word with =()= |
| =sd{char}= | [S]urround [D]elete                | =sd'= → remove surrounding ='= |
| =sr{old}{new}= | [S]urround [R]eplace          | =sr)"= → change =()= to =""= |

*** Detailed examples

*Add:* =sa= + motion/text object + surrounding character

- =saiw)= — add =()= around the inner word under cursor
- =sa2w"= — add ="= around the next 2 words
- =saip}= — add ={}= around the inner paragraph
- =sa$)= — add =()= from cursor to end of line

*Delete:* =sd= + character to find and remove

- =sd"= — find surrounding ="= and delete them
- =sd)= — find surrounding =()= and delete them
- =sdt= — find surrounding HTML tag and delete it

*Replace:* =sr= + old character + new character

- =sr)"= — change =()= to =""=
- =sr"'= — change =""= to =''=
- =sr}>= — change ={}= to =<>=

*** Visual mode

In visual mode, select text first, then =sa{char}= wraps the
selection:

1. =viw= — select inner word
2. =sa"= — surround selection with double quotes

** mini.operators — Custom Operators

mini.operators adds new operator commands that work with motions
and text objects, just like =d= and =c=.

| Prefix        | Operator    | Example                              |
|---------------+-------------+--------------------------------------|
| =g={motion}=  | Evaluate    | =g=ip= evaluate inner paragraph     |
| =gR{motion}=  | Replace     | =gRiw= replace word with register   |
| =ge{motion}=  | Exchange    | =geiw= on two words to swap them    |
| =gm{motion}=  | Multiply    | =gmiw= duplicate the word           |
| =gs{motion}=  | Sort        | =gsip= sort lines in paragraph      |

*** Replace workflow

Replace is the most commonly used operator. It pastes from the
register over the target, without overwriting the register:

1. =yiw= — yank a word
2. Move to another word
3. =gRiw= — replace that word with the yanked text
4. Move to another word → =gRiw= again (register still has original yank)

This is better than =diw= then =P= because =d= would overwrite the
register.

*** Exchange workflow

Exchange swaps two text regions:

1. =geiw= — mark first word for exchange
2. Move to another word
3. =geiw= — the two words are swapped

*** Sort

=gsip= sorts all lines in the current paragraph alphabetically.
=gs5j= sorts the next 5 lines.

** Julia IDE

Your Neovim config provides a complete Julia development environment

*** LSP

*** REPL — iron.nvim

The Julia REPL is fully working via iron.nvim. It launches Julia with

falls back if not installed). The REPL opens in a vertical split on

**** REPL control

| Key               | Action                               |
|-------------------+--------------------------------------|
| =<leader>rs=       | [R]epl [S]tart (open Julia REPL)    |
| =<leader>rr=       | [R]epl [R]estart                    |
| =<leader>rf=       | [R]epl [F]ocus (jump to REPL)       |
| =<leader>rh=       | [R]epl [H]ide                       |
| =<leader>rq=       | [R]epl exit (quit)                  |

**** Sending code to the REPL

| Key               | Action                               |
|-------------------+--------------------------------------|
| =<leader>rl=       | Send current [l]ine                 |
| =<leader>rp=       | Send current [p]aragraph            |
| =<leader>ra=       | Send [a]ll (entire file)            |
| =<leader>ru=       | Send [u]ntil cursor                 |
| =<leader>rm=       | Send [m]otion (Normal mode)         |
| =<leader>rm=       | Send selection (Visual mode)        |
| =<leader>r<CR>=    | Send carriage return to REPL        |
| =<leader>r<space>= | Interrupt (like Ctrl-C in terminal) |

**** Typical Julia session

1. =nvim src/main.jl= → open a Julia file (LSP starts loading)
2. =<leader>rs= → Julia REPL opens in vertical split on the right
3. Write a function in the left buffer
4. =<leader>rp= → sends the paragraph (function) to the REPL
5. =<leader>rf= → focus the REPL, test interactively
6. =<leader>rh= → hide the REPL, focus returns to code
7. Edit the function → =<leader>rl= → re-send just the changed line
8. Revise.jl auto-reloads changed function definitions

**** About Revise.jl

The REPL startup command tries =using Revise= inside a try/catch. If
Revise is installed, function redefinitions are tracked automatically:
change a function in your source file, send it to the REPL, and the
new version takes effect immediately — no REPL restart needed. If
Revise is not installed, a warning prints and the REPL works normally
without tracking.

Install Revise if you don't have it yet:

#+BEGIN_SRC bash
julia -e 'using Pkg; Pkg.add("Revise")'
#+END_SRC
